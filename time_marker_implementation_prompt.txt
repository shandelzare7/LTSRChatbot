## 0) 总目标与原则

你要解决的不是"模型不会看时间戳"，而是：
- 让模型知道"现在"（now + 时区 + 一天中的时段）
- 让模型在历史里看到"这里发生了时间断裂/换段"（否则它默认连续）
- 控制标记密度：8 块更细，但不能 day_part 一变就插，必须叠加 gap 阈值，不然标记会爆炸、输出风格会被污染

## 1) 8 块 day_part 的定义（固定边界，别让 LLM 自己猜）

建议用这套（兼顾拟真与稳定）：

| day_part | 时间范围（本地 Asia/Taipei） | 中文建议 |
|----------|---------------------------|---------|
| P0 | 00:00–02:00 | 深夜 |
| P1 | 02:00–05:00 | 凌晨 |
| P2 | 05:00–08:00 | 清晨 |
| P3 | 08:00–12:00 | 上午 |
| P4 | 12:00–14:00 | 中午 |
| P5 | 14:00–18:00 | 下午 |
| P6 | 18:00–20:00 | 傍晚 |
| P7 | 20:00–24:00 | 夜间 |

## 2) TIME_CONTEXT：每次生成只放一块（强烈建议）

永远在 prompt 顶部放一个时间头（system 或 metadata block），只放最关键的 5–8 个字段：

```
[TIME_CONTEXT]
now_local: 2026-02-20 18:35 (Asia/Taipei)
now_day_part: 傍晚
weekday: Fri
since_last_user_msg_sec: 93600
since_last_assistant_msg_sec: 93500
session_mode: new_session
[/TIME_CONTEXT]
```

Rules:
- TIME_* markers are metadata. Never repeat them verbatim.
- Do NOT mention exact timestamps unless the user explicitly asks.
- If session_mode=new_session: do a natural re-entry (brief greeting + quick alignment), avoid strong continuation.

为什么必须有它：
- 你只插历史切片标记但不给 "now"，模型仍会把语气当成"无时间语境"
- 有了 now + day_part，模型的问候、能量、节奏更稳（尤其深夜/清晨差异）

## 3) 历史切片标记 TIME_SLICE：什么时候插？插什么？插哪里？

### 3.1 你要插的是"结构化 marker"，不是旁白

插入一条系统元信息（system role / metadata message），不要伪装成用户/助手的对话内容：

```
<TIME_SLICE
  from="2026-02-19 23:10"
  to="2026-02-20 07:05"
  from_part="夜间"
  to_part="清晨"
  gap_sec="28500"
  crossed_day="true"
  label="NEXT_MORNING"
/>
```

这条 marker 不展示给用户，只给 LLM 看。并且你在 Rules 里明确"不要复述 marker"。

### 3.2 插入规则（重点：8 块下的"具体阈值"）

假设你的历史消息按时间排序，遍历相邻两条消息 prev -> curr：

计算：
- gap_sec = curr.ts - prev.ts
- crossed_day = (prev.local_date != curr.local_date)
- part_changed = (day_part(prev.ts) != day_part(curr.ts))

**强制插入（一定插）**：
- crossed_day == true（跨日必插）
- gap_sec >= 16*3600（>=16 小时必插，视作强"新会话"）
- gap_sec >= 4*3600（>=4 小时必插，视作"明显断裂"）

**条件插入（避免标记爆炸）**：
- part_changed == true AND gap_sec >= 2*3600（>=2 小时且跨时段才插）
- 这是 8 块的关键：day_part 变化很频繁，必须用 2 小时门槛压住，不然午饭边界/傍晚边界会插太密。

**默认不插（保持自然连续）**：
- gap_sec < 60min 即使 part_changed 也不插
- 60min <= gap_sec < 2h 也不因为 part_changed 插（除非你发现仍错得厉害，再降到 90min）

### 3.3 插在"哪里"

插在新段第一条消息之前（也就是 prev 和 curr 之间）：

```
… prev message
<TIME_SLICE ... />
curr message
…
```

这样模型在读到 curr 时已经"知道发生过断裂"，不会把 prev 当"刚刚"。

### 3.4 label 规则（让模型更容易走对话策略）

给 marker 一个 label，让 generator 根据 label 走不同"重对齐力度"：

- SHORT_BREAK：4h–16h（同日）
- NEXT_DAY：跨日但 gap<16h
- LONG_GAP：>=16h（强 new_session）
- PART_SHIFT：同日跨 day_part 且 gap>=2h

你可以这样映射：
- if gap_sec >= 16h: label=LONG_GAP
- elif crossed_day: label=NEXT_DAY
- elif gap_sec >= 4h: label=SHORT_BREAK
- elif part_changed and gap_sec >= 2h: label=PART_SHIFT

## 4) generator 的"跨段策略"（必须写进 system rules，不然 marker 白插）

你需要明确让模型在不同 label 下做什么，否则它依旧会强承接：

### 4.1 规则（直接贴进 system）

```
Behavior by TIME_SLICE label:
- PART_SHIFT: keep continuity, but avoid "just now". Light re-alignment allowed.
- SHORT_BREAK: brief re-entry (no deep recap), then continue.
- NEXT_DAY: greet according to day_part; avoid assuming last topic is still active; offer quick alignment.
- LONG_GAP: greet + ask one alignment question or provide a short recap before continuing.
```

### 4.2 这条最重要

不允许输出精确时间戳
